## Commands

### Backend

```bash
uv sync                                                                      # install dependencies
docker compose up -d db                                                     # start Postgres + pgvector
uv run alembic upgrade head                                                 # apply migrations (required before first run)
uv run uvicorn --app-dir src compliance_doc_assistant.main:app --reload     # run the API (http://127.0.0.1:8000/docs)
uv run ruff format .                                                        # format
uv run ruff check .                                                        # lint
uv run pytest                                                               # run all tests
```

### Frontend

```bash
cd frontend
npm install
npm run dev      # http://localhost:3000
npm run lint
npm run build
```

CI (`.github/workflows/ci.yml`) has four jobs: `backend-test` (spins up a `pgvector/pgvector:pg16` service container, creates a separate `compliance_doc_assistant_test` database, then runs `ruff format --check`, `ruff check`, `alembic upgrade head`, `pytest` — with embeddings and answer generation stubbed, see below), `pgvector-smoke` (its own `pgvector/pgvector:pg16` service container, runs migrations, then `pytest pgvector_smoke_test/ -v` — the one place that loads the real `sentence-transformers` model and runs a real cosine-distance nearest-neighbor query), `ollama-smoke` (a `pgvector/pgvector:pg16` **and** an `ollama/ollama` service container together, pulls `qwen2.5:1.5b`, then `pytest ollama_smoke_test/ -v` — the one place that calls a real LLM end-to-end and gets a real generated answer, closing the live-verification gap left by not having a funded `ANTHROPIC_API_KEY` in dev; mirrors evalops-dashboard's `postgres_smoke_test`/`ollama_smoke_test` pattern), and `frontend` (`npm ci`, `npm run lint`, `npm run build`).

`tests/conftest.py` has an autouse `_stub_embeddings` fixture that monkeypatches `compliance_doc_assistant.ingestion.embed_texts` with a deterministic fake (distinct vectors per distinct text, no real model load) — this keeps the default `pytest` run fast and network-free. `tests/test_questions.py` has its own file-scoped autouse fixture overriding the `get_generation_client` FastAPI dependency with a `FakeGenerationClient`, the same `app.dependency_overrides` pattern evalops-dashboard used for its LLM judge — generation is only relevant to that one test file, unlike embeddings which every document-touching test needs, so it isn't a global conftest fixture. `pgvector_smoke_test/` and `ollama_smoke_test/` both live outside `tests/` specifically so neither stub applies there.

Unlike `evalops-dashboard`, there is **no SQLite fallback** — `pgvector`/`vector` columns require a real Postgres with the extension enabled, so `docker compose up -d db` (or CI's service container) is required before running migrations or tests at all. `docker/init-test-db.sql` creates the dedicated `compliance_doc_assistant_test` database (and enables the extension on it) automatically on first container startup.

Database URL is configurable via `COMPLIANCE_DATABASE_URL` (defaults to a local `compliance_doc_assistant` Postgres database matching `docker-compose.yml`); `COMPLIANCE_TEST_DATABASE_URL` (defaults to `compliance_doc_assistant_test`) is what `tests/conftest.py` forces the app to use, guarded by a `RuntimeError` if the resolved database name doesn't contain `test`.

## Architecture

FastAPI + SQLModel app under `src/compliance_doc_assistant/`, mirroring evalops-dashboard's flat-modules-plus-routers style:

- `database.py` — single SQLAlchemy `engine`, built from `COMPLIANCE_DATABASE_URL`. No SQLite branching (unlike evalops-dashboard) since Postgres is mandatory here.
- `models.py` — all SQLModel table models plus their `*Create`/`*Read` DTOs, colocated. `User`/`AuthSession` (Milestone 2), `Document`/`Chunk` (Milestone 3; `Chunk.embedding` added in Milestone 4). `DocumentStatus` and `ChunkRead`/`DocumentDetailRead` DTOs live here too. `Document.status` is forced to a plain `sa.String(length=20)` rather than a native Postgres ENUM, same rationale as evalops-dashboard's `User.role` — new status values need no migration (Milestone 4 added `EMBEDDED` as the new terminal success status this way, no migration required). `Chunk.embedding` (`pgvector.sqlalchemy.Vector(EMBEDDING_DIMENSIONS)`) is never client-submittable — lives only on `Chunk`, not `ChunkBase`/`ChunkRead`, same server-controlled-results placement as `Document.status`.
- `auth.py` — session-based auth (bcrypt hashing, opaque `secrets.token_urlsafe` tokens, cookie or `Authorization: Bearer`), direct port of evalops-dashboard's pattern minus RBAC/rate-limiting (deliberately deferred, see `LEDGER.md` Milestone 2).
- `chunking.py` — pure function `split_into_chunks(text, chunk_size_words, overlap_words) -> list[ChunkDraft]`. No DB, no model, no I/O. "Words" (whitespace-split) stand in for tokens — there's still no real tokenizer wired up (the sentence-transformers model does its own tokenization internally, but chunk sizing doesn't call into it). Stops emitting windows once one reaches the end of the text, rather than always advancing by a fixed stride — otherwise a stride that doesn't evenly divide the text length produces a tiny, almost-entirely-duplicate trailing chunk (caught by `test_produces_overlapping_windows` during Milestone 3).
- `embeddings.py` — `EMBEDDING_DIMENSIONS = 384` (matches the migration's `Vector(384)` column, and `all-MiniLM-L6-v2`'s real output size). `_get_model()` lazily loads the `sentence-transformers` model behind an `lru_cache(maxsize=1)` singleton — never at import time. `embed_texts(texts) -> list[list[float]]` is the only public entry point.
- `ingestion.py` — `extract_text` dispatches on file extension (`.txt` decoded directly, `.pdf` via `pypdf`, wrapping `PdfReadError` into a domain `InvalidPdfError` rather than leaking the raw library exception). `ingest_document` is a documented deviation from the pure-module pattern (like `auth.py`'s session functions): it does real DB I/O directly — persists the `Document`, calls `chunking.split_into_chunks`, calls `embeddings.embed_texts` on the chunk contents, persists the `Chunk` rows with their embeddings attached, and flips `Document.status` from `pending` straight to `embedded` (chunking and embedding happen synchronously in one request, so there's no externally-visible "chunked but not yet embedded" intermediate state).
- `retrieval.py` — `retrieve_relevant_chunks(session, document_id, query_embedding, top_k)` does a real pgvector nearest-neighbor query (via `Chunk.embedding.cosine_distance(...)`, using the HNSW index), scoped to one document and excluding null embeddings, returning `RetrievedChunk` (chunk + similarity score, `1 - distance`) ordered by relevance. Another documented DB-I/O module, same rationale as `ingestion.py` — the ranking has to happen in SQL to use the index, not by pulling every chunk into Python.
- `generation.py` — swappable-provider RAG generation, structured exactly like evalops-dashboard's `llm_judge.py`: a `GenerationClient` Protocol (`provider: str`, `generate(question_text, chunks) -> GenerationResult`), `AnthropicGenerationClient` (default; lazy client, `ANTHROPIC_API_KEY` checked on first use, raises a clear `503` if missing, model configurable via `ANTHROPIC_GENERATION_MODEL` falling back to `claude-haiku-4-5-20251001`), `OllamaGenerationClient` (opt-in via `GENERATION_PROVIDER=ollama`, talks to `OLLAMA_HOST` falling back to `http://127.0.0.1:11434`, model via `OLLAMA_GENERATION_MODEL` falling back to `qwen2.5:1.5b`). `get_generation_client()`/`GenerationClientDep` is a real FastAPI dependency (unlike `embed_texts`, which is called directly) specifically so tests can override it via `app.dependency_overrides`, same mechanism `auth.py`'s dependencies use. `build_generation_prompt` is a pure function shared by both providers — only the client/response-parsing code differs. Citations aren't self-reported by the model (no forced tool-choice/structured output, unlike the judge's scoring) — every retrieved chunk that was included in the prompt becomes an `AnswerCitation`, in retrieval rank order; the model is asked to reference chunk numbers in prose, but the formal citation records are mechanically derived from what was retrieved, not from asking the model which ones it "actually used." Kept deliberately simple for v1 — see `LEDGER.md` Milestone 6 if this ever needs to change.
- `confidence.py` — pure `assess_confidence(scores: list[float]) -> ConfidenceResult` (`needs_review: bool`, `reason: str | None`), same shape as evalops-dashboard's `scoring.py`. Assumes `scores` is pre-sorted descending (as `retrieval.retrieve_relevant_chunks` already returns them) — doesn't sort itself. Flags for review when there are no matches at all, when the top match's similarity is below `LOW_SIMILARITY_THRESHOLD` (0.35, calibrated from real `all-MiniLM-L6-v2` scores observed manually in Milestones 5-6: genuinely relevant matches scored ~0.51-0.64, an unrelated one scored ~0.19 — a starting point, not tuned against a real eval set), or when the gap between the top two matches is below `AMBIGUOUS_GAP_THRESHOLD` (0.03, meaning it's unclear which chunk is authoritative). The low-similarity check is checked first and wins if both conditions would apply.
- `review.py` — the confidence decision's persistence layer, mirroring the `scoring.py` (pure) / `evaluations.py` (persist) split: `maybe_create_review_flag(session, answer)` creates a `ReviewFlag` row only if `answer.needs_review`, `list_pending_flags(session, owner_id)` joins `ReviewFlag → Answer → Question → Document` scoped to one owner, `get_owned_review_flag` is the same anti-enumeration ownership-check pattern as `get_owned_document`, `resolve_review_flag` sets status/reviewed_by/reviewed_at.
- `main.py` — app instance, lifespan hook that seeds a demo user via `seed.py`, and `GET /health`.

- `routers/documents.py` — `POST /documents` (upload, enforces a 10MB size limit before calling `ingestion.ingest_document`, translates its domain errors to 415/422), `GET /documents` (list, scoped to `current_user`), `GET /documents/{id}` (detail with chunks). All three require auth (`CurrentUser`). Documents are scoped per-owner: a document ID belonging to another user 404s (not 403), same anti-enumeration principle already established for auth in Milestone 2 — `get_owned_document` is the shared helper for that check, exported (not module-private) specifically so `routers/questions.py` can reuse it rather than duplicating the ownership check.
- `routers/questions.py` — `POST /documents/{id}/questions`: looks up the document via `get_owned_document` (404 for other users' documents), 422s if `Document.status != EMBEDDED` (defensive — every document reachable through the API today is always `embedded` by the time it's persisted, but this guards against a future async ingestion pipeline reintroducing a real gap), persists the `Question`, embeds the question text, retrieves ranked chunks, calls `generation_client.generate(...)` (skipped entirely if there are no matches — `_generate_answer` short-circuits to a canned "not enough information" answer rather than prompting a model with empty context), runs `confidence.assess_confidence` on the retrieval scores, persists the `Answer` (with `needs_review`/`confidence_reason` set) + `AnswerCitation` rows, calls `review.maybe_create_review_flag`, and returns the answer with citations — `needs_review`/`confidence_reason` are on `AnswerBase` so they're already in the response with no extra wiring. This endpoint's response shape changed from Milestone 5 (`question` + raw `matches`) to Milestone 6 (`question` + generated `answer` with `citations`) — a deliberate evolution of the same endpoint across milestones, not a new one; there was never an external consumer of the Milestone 5 shape to break.
- `routers/review.py` — `GET /review-flags` (pending flags for `current_user`'s documents only, via `review.list_pending_flags`), `POST /review-flags/{id}/resolve` (body `{"status": "resolved"|"dismissed"}`, rejects `"pending"` as a target with 422, 404 for another user's flag via `get_owned_review_flag`).

### Migrations

Schema changes go through Alembic (`alembic/versions/`), not app startup. `alembic/env.py` imports `compliance_doc_assistant.models` for autogenerate metadata and reads `COMPLIANCE_DATABASE_URL` the same way the app does.

## Frontend

Next.js (TypeScript, App Router) app in `frontend/`, kept as a fully separate package (own `package.json`/lockfile, no shared code with the backend) — a deliberate portfolio decision to diversify beyond evalops-dashboard's server-rendered Jinja2/zero-client-JS approach.

**Auth architecture (Milestone 8 decision):** Bearer token in `localStorage`, not cookies — the frontend is a separately-hosted app talking to the backend cross-origin, and a Bearer header sidesteps the SameSite/credentialed-CORS complexity a cross-origin cookie would need. Backend gained `CORSMiddleware` (`FRONTEND_ORIGINS` env var, default `http://localhost:3000`) with `allow_credentials` deliberately left off, since the Authorization header isn't subject to the browser's credentialed-request restrictions the way cookies are. Tradeoff, stated plainly: the token is readable by any injected script (XSS risk) — acceptable for a portfolio demo, not for handling real sensitive data. The alternative (`SameSite=None; Secure` cookie, or a same-origin Next.js proxy) would be more correct but adds real complexity for a demo project; noted here in case this ever needs to change.

- `lib/api.ts` — `apiFetch` wrapper attaches `Authorization: Bearer <token>` from `localStorage` (`getStoredToken`/`setStoredToken`) to every request; typed functions per backend endpoint (`login`, `getCurrentUser`, `logout`, `listDocuments`, `getDocument`, `uploadDocument`, `askQuestion`, `listReviewFlags`, `resolveReviewFlag`). `ApiError` carries the HTTP status and the backend's `detail` message.
- `lib/types.ts` — TypeScript interfaces mirroring the backend's Pydantic response DTOs (`DocumentRead`, `DocumentDetailRead`, `ChunkRead`, `QuestionAnswerRead`, `AnswerWithCitationsRead`, `ReviewFlagRead`, etc.) — kept in sync by hand, not generated.
- `lib/auth-context.tsx` — `AuthProvider`/`useAuth()`, a React context holding `token`/`user`/`isLoading`. On mount, hydrates `user` from the stored token via `GET /me` (not just trusting the stored token blindly — clears it if the backend rejects it as expired/revoked). The hydration effect is written to avoid ever calling `setState` synchronously in the effect body (routes every state update through a `.then()`/`.catch()`/`.finally()` callback) — a newer `react-hooks/set-state-in-effect` lint rule flags the more obvious version of this pattern.
- `components/RequireAuth.tsx` — client-side route guard wrapping protected pages; redirects to `/login` if `!isLoading && !token`. There's no Next.js middleware for this since the token lives in `localStorage`, which server-side middleware can't read.
- `components/NavBar.tsx` — shows "Log in" when unauthenticated, or Documents link + username + "Log out" when authenticated.
- `app/login/page.tsx`, `app/documents/page.tsx` (list + `DocumentUploadForm`), `app/documents/[id]/page.tsx` (detail: `AskQuestionForm` + a running list of `AnswerCard`s, each showing the answer, citations, and a review-flag banner with Mark resolved/Dismiss buttons when `needs_review` is true).
- `components/AnswerCard.tsx` — the review-flag banner needs the `ReviewFlag.id` to call `POST /review-flags/{id}/resolve`, but `QuestionAnswerRead` doesn't nest it (only `needs_review`/`confidence_reason` are on the answer). The document detail page cross-references by calling `GET /review-flags` and matching on `answer_id` client-side rather than extending the backend response — kept scope to frontend-only for this milestone.
- No dedicated global "all my review flags" page in v1 — flags surface inline on whichever document's Q&A they belong to, matching the plan's "review-flag banner" framing. A cross-document flags list would need `ReviewFlagRead` to carry document context; noted as a possible follow-up, not built.
