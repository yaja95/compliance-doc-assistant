CI on the PR — both the `push` run ([`29215403805`](https://github.com/yaja95/compliance-doc-assistant/actions/runs/29215403805)) and the `pull_request` run ([`29215410252`](https://github.com/yaja95/compliance-doc-assistant/actions/runs/29215410252)) passed (`backend-test` + `frontend` jobs).
- **Shipped:** `User`/`AuthSession` SQLModel tables + migration; session-based auth (`auth.py`) — bcrypt password hashing, opaque `secrets.token_urlsafe` session tokens (not JWT, matching evalops-dashboard's rationale), `HttpOnly`/`SameSite=Lax` cookie or `Authorization: Bearer` support, 7-day session lifetime; `POST /auth/login`, `POST /auth/logout`, `GET /me` (the first protected route, demonstrating auth is enforced); idempotent demo-user seeding on startup via a `seed.py` lifespan hook (`SEED_USER_PASSWORD` env var, documented dev-only fallback). Verified live against the full `docker compose up --build` stack: real login returns a cookie + token, `GET /me` correctly 401s without auth and 200s with the returned token.
- **Deliberately deferred (scope decision, not an oversight):** RBAC (`role` field, admin-gated `POST /users`) and login rate limiting — both came *after* evalops-dashboard's initial auth milestone there too (Milestones 13–14, not 10), so this repo follows the same incremental order rather than front-loading everything into one milestone. There is currently only one way to create a user: the seeded `demo` account. `POST /users` will need to be added, gated somehow, once multi-user support is actually needed.
