## Milestone 3 — Document ingestion and chunking
- **Status:** Done
- **Merged:** 2026-07-13T01:03:21Z, squash commit [`de52b95`](https://github.com/yaja95/compliance-doc-assistant/commit/de52b95835c35494b24819526ca1701cf1b3a315) on `main`, PR [#2](https://github.com/yaja95/compliance-doc-assistant/pull/2). Verified via `gh pr view 2` (state MERGED) and `git log`/`git pull` against `origin/main` directly.
- **Post-merge checks re-run against `main`:** `ruff format --check .`, `ruff check .` clean; `alembic upgrade head` applies cleanly; `pytest` 29/29 passing. CI on the PR — both the `push` run ([`29216286247`](https://github.com/yaja95/compliance-doc-assistant/actions/runs/29216286247)) and the `pull_request` run ([`29216297886`](https://github.com/yaja95/compliance-doc-assistant/actions/runs/29216297886)) passed (`backend-test` + `frontend` jobs).
- **Shipped:** `Document`/`Chunk` SQLModel tables + migration (no `embedding` column yet — Milestone 4); `chunking.py` — pure fixed-size chunking (~500 words/~50-word overlap, "words" standing in for tokens until a real tokenizer arrives with embeddings), no DB/model/I/O; `ingestion.py` — `.txt`/`.pdf` text extraction via `pypdf` (malformed-PDF errors wrapped into a domain `InvalidPdfError` rather than leaked), `ingest_document` persists the `Document` + `Chunk` rows and flips status `pending` → `chunked` (a documented pure-module deviation, same as `auth.py`'s session functions); `routers/documents.py` — `POST /documents` (10MB limit, 415/422 error mapping), `GET /documents` (owner-scoped list), `GET /documents/{id}` (detail with chunks), all requiring auth, with another user's document 404ing rather than 403ing (anti-enumeration, matching Milestone 2's auth). Verified live against the full `docker compose up --build` stack: real upload of a `.txt` file returns 201 chunked, list/detail reflect it, unsupported type → 415.
- **Bugs found and fixed during implementation, not planning:** (1) *Chunking off-by-stride bug* — the initial loop always advanced by a fixed stride regardless of where the current window ended, so when the stride didn't evenly divide the text length it emitted a tiny, almost-entirely-duplicate trailing chunk; fixed by stopping once a window reaches the end of the text (caught by `test_produces_overlapping_windows`, a new reference case worth watching for in future windowing/pagination code). (2) *Test-only cross-user cookie leakage* — a test simulating two logged-in users on one `TestClient` failed because `resolve_token_from_request` checks the `session_token` cookie before the `Authorization` header, and `TestClient` shares one cookie jar across logins, so the most recently logged-in user's cookie silently overrode explicit Bearer headers set for an earlier-logged-in user. Not a production bug — real clients don't send conflicting cookies and headers for different identities — but a real pitfall for any future test simulating multiple concurrent identities on one `TestClient`; fixed by clearing the client's cookies right after capturing each login's token.

- **Shipped:** `Question` model + migration; `retrieval.py` — `retrieve_relevant_chunks(session, document_id, query_embedding, top_k)`, a real pgvector nearest-neighbor query via `Chunk.embedding.cosine_distance(...)` (uses the HNSW index), scoped to one document, excludes chunks with null embeddings, returns ranked `RetrievedChunk` (chunk + `1 - distance` similarity score) — a documented DB-I/O module like `ingestion.py`, since ranking has to happen in SQL to use the index; `POST /documents/{id}/questions` embeds the question text, calls retrieval, persists the `Question`, and returns ranked matches with scores — no answer generation yet (Milestone 6). `get_owned_document` promoted from a module-private helper in `routers/documents.py` to a shared, exported one so `routers/questions.py` reuses the same ownership check (404, not 403, for another user's document) instead of duplicating it. Verified live against the full `docker compose up --build` stack: real upload → real question → correctly ranked match with a sensible similarity score.
- **Caught during implementation, not planning:** autogenerate proposed *dropping* the Milestone 4 HNSW index in this migration — it's created via raw `op.execute`, not declared in SQLAlchemy metadata, so autogenerate sees "index exists in DB but not in models" and treats it as removed. Caught by reading the generated migration before applying it (not by a test); the erroneous `op.drop_index(...)` line was removed and the index confirmed still present in Postgres afterward. Worth remembering for any future migration touching `chunk`: always read autogenerate output before running it when a raw-SQL-created index/constraint exists on the same table.
